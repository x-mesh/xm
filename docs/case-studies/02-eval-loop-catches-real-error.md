# 케이스 2 — 자기채점을 믿지 마라: eval 루프가 실오류를 잡다

**날짜**: 2026-07-04 · **소요**: 채점→수정→재채점 약 20분 · **심판**: 3-judge (standard 2 + adversarial 1)

## 상황

크로스벤더 브레인스토밍 산출물(전략 아이디어 24개/9테마 JSON)을 만들고, 만든 쪽(리더)이
자기채점 **8.1/10**을 매겼다. 그 직후 `/xm:eval score`로 독립 3-judge 패널에 넘겼다.

## 패널이 잡은 것

**v1 판정: 6.99/10 — 통과 기준(7.0)에 0.01 미달, FAIL.**

핵심: **3명의 심판이 서로 독립적으로 같은 실오류를 적발**했다 — 요약에 "22 ideas"라고 두 곳에
적혀 있는데 실제 배열을 세면 24개. 한 번 잘못 센 값이 검증 없이 전파된 것. adversarial 심판의
표현: *"오타가 아니라 한 번 계산된 잘못된 값이 그대로 전파됐다는 뜻"*.

자기채점(8.1)과 패널(6.99)의 격차 1.1의 대부분이 만든 사람 눈에 안 보인 이 오류에서 나왔다.

## 수정 → 재채점

- 22→24 수정 + `corrections` 필드에 수정 이력 기록(누가 언제 왜)
- **신규 패널**로 재채점(같은 심판 재사용 시 앵커링 편향) → **7.82/10 PASS** (Δ+0.83)
- 재채점의 adversarial 심판은 한 단계 더 깊은 지적을 냈다: "요약 JSON만으로는 벤더 귀속을
  검증할 수 없다"(저자 귀속 환각 리스크) → 원본 응답 디렉토리 참조(`cross_vendor.run_ref`)를
  결과 스키마의 **필수 필드로 승격**시켰다. 심판의 지적이 제품 스키마를 바꾼 것.

## 증거 사슬

- v1 판정: `.xm/eval/results/20260704-144231-score.json` (판정 근거·심판별 rationale 포함)
- v2 판정: `.xm/eval/results/20260704-144752-score.json` (v1 대비 delta, adversarial 지적 처분 포함)
- 수정 이력: `.xm/op/brainstorm-2026-07-04-xm-differentiation-strategy.json`의 `corrections[]`
- 스키마 변경: `x-op/skills/op/references/x-op-result-persistence.md` § Cross-vendor provenance

## 이것이 보여주는 것

1. **자기채점은 체계적으로 낙관적이다** — 같은 산출물, 같은 기준, 8.1 vs 6.99.
2. **경계값 판정이 의미를 가진다** — 0.01 차 FAIL이 "대충 통과"로 뭉개지지 않고 수정을 강제했다.
3. **adversarial 심판의 지적이 일회성 코멘트가 아니라 스키마(제도)로 남는다.**
