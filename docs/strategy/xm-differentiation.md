## 2. 경쟁 환경 + 플랫폼 충돌 분석  **[v2 강화: ultra 충돌 명시]**

- **gstack**: 시각적 end-to-end 제품 개발(browse/qa/design/ship). → 정면승부 안 함.
- **term-mesh**: 실시간 에이전트 팀·터미널 UX. → 영역 안 겹침.
- **Claude Code 네이티브 흡수**: Agent/Workflow tool(결정적 fan-out), **`/code-review ultra`(클라우드 멀티에이전트 *리뷰*)**.

**핵심 충돌 (v1 누락, 패널 만장일치 지적):**
`/code-review ultra`는 **멀티에이전트 *검증* 기능 그 자체**다. v1은 "fan-out만 흡수되고
검증 계층은 방어 가능"이라 가정했으나, 플랫폼은 이미 **검증을 흡수 중**이다.
→ 단, ultra가 검증하는 대상은 **코드 diff**다. xm이 검증해야 할 대상은
**"어떤 전략/프롬프트가 더 나은 산출물을 냈는가"** — 즉 *오케스트레이션 선택* 이다.
이 **검증 대상의 차이**가 v2 차별화의 축이다.

## 3. 개정된 명제 (THE BET v2)  **[v2 변경: 해자 축소·재정의]**

> v1 명제("검증 가능한 오케스트레이션 규율")는 살아있으나, 가장 싼 기둥(비용 actuals)이
> 구조적으로 막혀 있고 검증 기둥이 플랫폼과 충돌한다. 따라서 해자를 좁힌다.

**xm의 방어 가능한 해자는 단일 능력이 아니라 세 요소의 *결합* 이다:**
1. **전략-인지 평가(strategy-aware eval)** — `/code-review ultra`가 안 하는 것:
   코드가 아니라 *전략·프롬프트 선택*을 커밋된 골든셋·인간 ground-truth로 채점.
2. **xm이 통제하는 계층에서의 정직한 비용 회계** — 서브에이전트/Workflow 실측은 포기하고,
   (a) 추정 기반 예산을 *추정이라 명시* 하여 전 경로 적용 + (b) **xm이 *외부* CLI를 직접
   스폰하는 경로(= x-panel 한정)에서만 실측 비용 캡처**(외부 CLI가 usage/credits를
   출력함 — 예: kiro `Credits: 1.10`. 검증 필요 가설). **주의: x-agent flow는 Workflow-tool
   서브에이전트 백엔드라 외부 CLI 경로가 아니며, ⛔ 차단 범주에 속한다(여기 포함 금지).**
3. **재현 가능한 결정 메모리** — x-trace가 재현 입력(prompt/seed/명령)까지 저장하도록
   확장해 "이 결정 재현" 명령을 진짜로 만든다.

**이 결합을 측정된 성과 델타로 입증** 하는 것이 베팅이다. 단일 능력은 모두 복제·흡수
가능하므로, 차별화는 **통합 + 규율 + 입증**에 건다.

## 4. 3대 베팅 (개정)

### Bet A — Strategy-aware "verified" (대상을 코드→전략으로 이동)  **[v2 변경]**
- **무엇**: x-op/x-agent 산출물을 커밋된 골든셋 + **인간 ground-truth**로 채점.
- **`/code-review ultra`와의 차별점(명시)**: ultra=코드 diff 리뷰 / Bet A=전략 선택 평가.
- **정직화**:
  - "기본 검증(verified by default)"은 **현재 상태가 아니라 90일 목표**다(현재는 opt-in
    `--verify`/`eval.auto`). 문서·UI에서 *현재 vs 목표*를 분리 표기.
  - KPI '오탐률'을 **시뮬레이터가 스스로 산출하지 않는다**(v1의 순환 결함). 오탐률은
    **인간 라벨 ground-truth** 로만 산출; 시뮬레이터는 임계값 *후보* 생성에만 사용.
- **선결 구축물(🟡→🟢 전환 필요)**: 프로덕션 골든셋(전략 2~3개; 현재 총 6개), 실행 가능한
  eval CLI, 통계적으로 유의한 라벨셋(현재 라벨 n=5 → 사전 검정력 계산이 요구하는 n).
- **공격받는 가정**: 사용자가 "전략이 더 낫다는 증명"에 비용·지연을 지불한다.

### Bet B — Cost honesty (actuals 캡처 포기, 정직한 추정 + CLI 경로 실측)  **[v2 전면 재정의]**
- **v1 폐기**: "모든 fan-out 실측 토큰 → budget guard"는 **서브에이전트 실측이 구조적
  차단**이라 불가. 이미 막힌 작업을 리스크-1순위로 둔 것이 v1 최악의 오류.
- **v2 대체**:
  - (a) **추정 기반 예산을 전 경로(x-op/x-agent 포함)로 확장** — 단 모든 수치에
    "estimate" 라벨을 박아 actuals로 오인되지 않게 한다.
  - (b) **외부 CLI 스폰 경로(x-panel 한정)에서만 실측 캡처** — 스폰한 *외부* CLI의
    **stdout/stderr 양쪽**에서 usage/credits 파싱(미구현·검증 필요; 일부 CLI는 credits를
    stderr로 출력 — 예: kiro). 여기는 Agent/Workflow 도구 제약을 받지 않는다.
    **x-agent flow는 Workflow 백엔드 = ⛔ 차단 범주이므로 이 분기에서 제외.**
  - (c) **legacy 미태깅 행 정화** — `computeTokenActuals`가 양성 `cost_source:'actual'`
    태그만 받도록 필터를 부정형→양성형으로 변경. **주의: 실측 캡처가 ⛔ 차단된 경로에서는
    이 변경이 actuals를 *정화*하는 게 아니라 *비움* → estimateTaskCost가 항상 추정으로
    폴백한다. 즉 (a) 추정-라벨 노선이 사실상 유일한 비용 경로임을 재확인하는 조치.**
- **차별화**: 경쟁 도구는 토큰을 태우지만 "추정인지 실측인지"조차 구분 안 해줌.
- **공격받는 가정**: CLI stdout usage 파싱이 모델 CLI 전반에서 안정적인가(미검증).

### Bet C — Discipline, *측정 가능하게*  **[v2 변경: 측정안 부착, 아니면 강등]**
- **v1 결함**: "규율=해자"라면서 KPI·측정법이 없어 반증 불가(패널 만장일치).
- **v2 측정안(필수)**: 동일 과제를 규율 자산 **on/off A/B**로 실행 →
  (재실행률, 리뷰-수정 루프 길이, 오탐 finding 비율)의 **성과 델타**를 측정.
  - 델타가 유의하면 → 1급 해자로 승격.
  - 델타가 없으면 → **Bet C를 해자에서 강등**, "내부 품질 관행"으로만 유지.
- **현실 인정**: 규율 자산(SKILL.md 규칙)은 문서라 복제 빠름. 해자는 규율 자체가
  아니라 "규율→측정된 결과"의 누적 증거다.

   x-eval로 수집·채점하는 배선**(Bet A가 x-agent 산출물도 대상으로 선언하므로 필수) + 인간 라벨.
3. **(Bet B 정직화)** legacy 행 정화(필터 양성형 전환) + 추정치 "estimate" 라벨 전파 +

4. **(Bet A 차별화)** `/code-review ultra` 대비 포지셔닝 문서화(코드 vs 전략 평가).
5. **(가정 검증)** 사용자 WTP(§8-2) — "전략 증명에 비용·지연 지불 의향"을 인터뷰/소규모
   설문으로 사전 검증(Bet A 전체의 전제이므로 구축 전 게이트).
6. **(Bet C 측정)** 규율 on/off A/B 하니스 1개 구축, 성과 델타 1차 측정.
7. **(재현 계층)** x-trace에 재현 입력(prompt/seed/명령) 저장 옵션 추가 → "재현 명령" 실현.

## 7. 성공/실패 기준 (반증 가능하게)  **[v2 변경: §5 의존성 충돌 해소]**
- **선결 게이트(명시)**: ① Bet B 실측 분기는 *step 1 PoC 통과 시에만* 활성. 미통과 시
  Bet B는 "추정 예산"으로 축소되며, 이는 *실패가 아니라 사전 정의된 분기*다
  (v1의 §6↔§5 충돌 해소). ② **Bet C의 '오탐 finding 비율' 지표는 Bet A의 인간
  ground-truth 라벨셋에 의존** → Bet A 라벨셋 미완 시 Bet C A/B는 (재실행률,
  리뷰-수정 루프 길이)만으로 측정(오탐 지표는 라벨셋 완성 후 추가).
- **성공(정량)**: 재실행률·예산 초과 사고 측정 감소; **Bet C A/B는 표본·검정력을 사전
  고정한 뒤**(효과크기 d≈0.5 가정 시 양군 각 n≈64, power 0.8, α 0.05 — 파일럿으로
  효과크기 재추정) 성과 델타 유의(p<0.05); 골든셋 라벨 상관은 **n을 늘리는 것 자체가
  목표가 아니라**(표본 수 도달 ≠ 유의성), 사전 검정력 계산이 요구하는 n에서 유의성 달성.
- **성공(정성)**: "왜 xm?"에 사용자가 "전략 선택이 측정으로 입증돼서"라 답.
- **실패**: 위 측정이 전부 무의미하거나, 사용자가 여전히 "그냥 fan-out 잘돼서"라 답.

### v2.2 → v2.3 패치 (크로스-벤더 계층 구현 반영 — 능력 구현됨, 성과 미측정)

> §1·§3.2·Bet B가 식별한 핵심 통찰 — **"외부 CLI 스폰 경로(x-panel 한정)는 Agent/Workflow
> 도구 차단을 벗어나는 유일한 경로"** — 를 토대로, 그 경로를 **재사용 가능한 크로스-벤더
> 검증 엔진으로 일반화**하고 라이프사이클 전 계층에 배선했다(PR1-PR7). 단 아래는 **능력
> (capability) 구현**이며, **성과(outcome) 입증이 아니다** — 이 구분은 Bet A/C·§7에 그대로 유효.

- **엔진 추출 (PR1)**: x-panel의 외부 CLI 스폰 경로를 `xm panel cross` / `invokeProviderText`
  (원시 stdout, 스키마 없음)로 재사용 가능한 primitive화. 새 엔진 코드는 이 1회뿐, 이후는 전부 재사용.
- **7개 소비자 배선 (PR2-PR7, 전부 SKILL-only 가산, opt-in `--cross-vendor`, <2벤더 시 loud
  단일벤더 폴백 L6)**:
  primitive(x-agent fan-out/broadcast) · 생성(x-solver candidates/hypotheses) ·
  계획(x-build consensus) · 심의(x-op debate/council) · 리뷰(x-review findings) ·
  평가(x-eval judges). 각 PR은 엔진이 자기 diff를 적대 검증하는 **dogfood**로 게이팅(매 PR 실버그 1-4건 발견·수정).
- **§1 자산 등급 갱신**: "외부 CLI 스폰 경로 실측 캡처(x-panel 한정)" 행이 가졌던 통찰이
  이제 **크로스-벤더 *검증* 능력으로는 🟢(구현·dogfood 검증)** 으로 승급. 단 **비용 *실측*

  멀티에이전트)가 **구조적으로 불가능한** 것(경쟁사 모델로 자기 작업 검증)을 xm이 **실제
  출시**했다 → 차별점이 문서 주장에서 **선적된 구조적 사실**로 이동.
- **여전히 미해결(정직)**: 이 크로스-벤더 능력이 **측정 가능한 더 나은 성과**를 내는지는
