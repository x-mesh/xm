# Korean Output Style (avoid AI-slop)

xm 전 스킬의 한국어 출력 톤 가이드. SKILL.md의 `### Korean output style` 섹션은 이 문서를 참조한다.

핵심: **좋아 보이는 말보다 확인 가능한 말을 남긴다.**

---

## 적용 범위

| 대상 | 적용 |
|---|---|
| LLM이 사용자에게 출력하는 한국어 응답 | ✅ |
| SKILL.md 내 사용자 출력 템플릿(한국어 부분) | ✅ |
| README.ko.md, docs/ 한국어 산문 | ✅ |
| SKILL.md 영문 본문 (LLM 지침) | ❌ — 영문 유지 (CLAUDE.md 규칙) |
| 코드, 명령어, 플래그, 파일 경로 | ❌ — 원문 유지 |
| CLI가 직접 찍는 짧은 라벨/헬프 (`✅`, `⚠`) | ❌ — 손대지 않음 |

도메인 용어 첫 등장 시 한글(원어). 예: 결론(verdict), 단계(phase).

---

## Before / After 25종

### 1. 가짜 균형 — `A뿐만 아니라 B까지`

의미 없이 항목을 두 개 묶어 균형감을 연출하지만, 실제 정보량은 하나뿐.

**Before**
```
x-solver는 단순히 문제를 분해할 뿐만 아니라, 전략적인 접근을 통해
체계적이고 구조적인 해결책까지 제시합니다.
```

**After**
```
x-solver는 문제를 4가지 전략(decompose / iterate / constrain / pipeline) 중
하나로 풀어냅니다.
```

→ 추상 형용사 제거, **수치 + 명사**로 교체.

---

### 2. 결론 없는 권고 — 균형·맥락 핑계

"상황에 따라 다르다" 류의 안전한 비결론. 사용자는 결정 못 함.

**Before**
```
프로젝트의 특성과 팀의 요구사항을 종합적으로 고려하여, 상황에 맞는
적절한 균형점을 찾는 것이 중요합니다.
```

**After**
```
모노레포면 decompose, 단일 버그면 iterate를 쓰세요. 잘 모르겠으면
classify가 알아서 추천합니다.
```

→ **구체적 분기 + fallback 명령**을 제시. "균형" 같은 단어 금지.

---

### 3. 강조 인플레이션 — 부사 누적

"매우 / 완벽하게 / 강력한 / 혁신적인" 같은 형용사·부사가 정보 없이 쌓임.

**Before**
```
매우 강력한 다중 에이전트 오케스트레이션을 통해 완벽하게 최적화된
혁신적인 워크플로우를 제공합니다.
```

**After**
```
4개 에이전트를 병렬로 띄워 평균 응답 시간을 1/3로 줄입니다.
```

→ 형용사 0개. **숫자 + 동사**만 남김.

---

### 4. 권고형 누적 — `~해야 합니다` 연쇄

한 문단에 `~해야 / ~할 필요가 / ~가 중요` 가 3번 이상. 우선순위가 안 보임.

**Before**
```
먼저 SKILL.md를 작성해야 합니다. 그리고 테스트를 추가할 필요가 있습니다.
또한 README를 업데이트하는 것이 중요합니다. 마지막으로 릴리스 노트를
작성해야 합니다.
```

**After**
```
순서:
1. SKILL.md 작성
2. 테스트 추가
3. README 업데이트
4. 릴리스 노트
```

→ 권고형 어미를 **명사구 리스트**로. 순서가 자연어보다 명확.

---

### 5. 영어식 직역 — `이러한 점에서 / ~함에 있어`

"In this regard / when it comes to" 직역체. 한국어로는 어색하고 늘어짐.

**Before**
```
이러한 점에서 x-build는 프로젝트를 관리함에 있어 매우 유용한 도구라고
할 수 있을 것입니다.
```

**After**
```
x-build는 프로젝트를 phase 단위로 관리합니다.
```

→ `이러한 점에서 / ~함에 있어 / ~라고 할 수 있을 것입니다` 모두 제거. **what it does**만 남김.

---

### 6. 추상 효과어 — 효율성 / 생산성 / 사용자 경험 / 품질

효과는 말하지만, 실제로 무엇이 달라지는지 보이지 않는 문장.

**Before**
```
x-build는 개발 생산성과 사용자 경험을 향상시키는 데 도움을 줍니다.
```

**After**
```
x-build는 빌드 설정을 한 파일로 묶고, 환경별 옵션을 자동으로 분리합니다.
```

→ "생산성 향상" 같은 추상어를 제거하고, 실제 동작으로 치환.

---

### 7. 가능성 남발 — `~할 수 있습니다 / ~가능합니다`

기능 소개문에서 자주 나오는 표현. 문장이 느슨해지고 책임이 흐려짐.

**Before**
```
이 옵션을 사용하면 로그를 더 쉽게 확인할 수 있습니다.
```

**After**
```
이 옵션을 켜면 에러 로그가 화면 하단에 바로 표시됩니다.
```

→ "할 수 있다"를 "무엇이 어떻게 바뀐다"로 변경.

---

### 8. 수동태 — `~됩니다 / ~제공됩니다 / ~처리됩니다`

행위자가 사라져서 문장이 기계적으로 보임.

**Before**
```
요청이 서버에서 처리되며 결과가 사용자에게 제공됩니다.
```

**After**
```
서버가 요청을 처리하고 JSON 응답을 반환합니다.
```

→ 주체를 되살림. "누가 무엇을 하는지"가 보이게 함.

---

### 9. 목차 예고문 과잉 — `먼저 살펴보고 / 다음으로 알아보겠습니다`

블로그·문서에서 흔한 AI식 안내문. 길지만 정보량은 낮음.

**Before**
```
먼저 x-build의 개념을 살펴보고, 다음으로 설치 방법과 사용 예시를
알아보겠습니다.
```

**After**
```
구성은 세 부분입니다.
- x-build 개념
- 설치 방법
- 사용 예시
```

→ 예고문을 목차로 압축.

---

### 10. "중요합니다" 엔딩

결론처럼 보이지만 실제 지침이 없음.

**Before**
```
보안 설정을 적절히 관리하는 것이 중요합니다.
```

**After**
```
운영 환경에서는 기본 토큰을 교체하고, 관리자 권한은 최소 인원에게만
부여합니다.
```

→ 가치 판단을 실행 기준으로 바꿈.

---

### 11. 포괄어 남용 — 다양한 / 여러 / 많은 / 폭넓은

넓어 보이지만 구체적인 대상이 없음.

**Before**
```
이 도구는 다양한 개발 상황에서 폭넓게 활용할 수 있습니다.
```

**After**
```
이 도구는 배포 전 점검, 장애 원인 추적, 릴리스 노트 작성에 사용할 수 있습니다.
```

→ "다양한"을 실제 사용처 목록으로 치환.

---

### 12. 근거 없는 강점 주장 — 업계 표준 / 널리 사용 / 뛰어난 성능

검증되지 않은 권위 표현. 신뢰보다 광고 느낌을 줌.

**Before**
```
이 방식은 많은 기업에서 사용하는 검증된 업계 표준입니다.
```

**After**
```
이 방식은 설정 파일을 분리하고, 배포 환경마다 다른 값을 주입할 수 있게 합니다.
```

→ 평판 주장을 기능 설명으로 바꿈. 필요하면 근거 링크나 수치를 붙임.

---

### 13. 동어 반복 — 효율적으로 관리하고 체계적으로 운영

비슷한 의미의 단어를 겹쳐 문장이 부풀어짐.

**Before**
```
프로젝트를 효율적으로 관리하고 체계적으로 운영할 수 있습니다.
```

**After**
```
프로젝트를 phase, task, release 단위로 나눠 관리합니다.
```

→ 비슷한 형용사를 제거하고, 분류 기준을 제시.

---

### 14. 부사어 패딩 — 기본적으로 / 일반적으로 / 상대적으로 / 전체적으로

문장을 안전하게 보이게 하지만, 정보 밀도를 떨어뜨림.

**Before**
```
기본적으로 이 방식은 상대적으로 안정적인 접근이라고 볼 수 있습니다.
```

**After**
```
이 방식은 배포 순서를 고정하므로 롤백 지점을 예측하기 쉽습니다.
```

→ 부사를 삭제하고, 안정적인 이유를 설명.

---

### 15. 문제 해결형 클리셰 — `기존 문제를 해결합니다`

어떤 문제가 어떻게 해결되는지 빠져 있음.

**Before**
```
이 기능은 기존 워크플로의 문제를 효과적으로 해결합니다.
```

**After**
```
이전에는 설정을 세 파일에서 수정해야 했습니다. 이제는 build.yml 한 곳만
바꾸면 됩니다.
```

→ "문제 해결"을 Before/After 구조로 바꿈.

---

### 16. 조건 없는 권고 — 필요하다면 / 상황에 따라 / 적절히

언제 실행해야 하는지 기준이 없음.

**Before**
```
필요하다면 캐시를 삭제하는 것이 좋습니다.
```

**After**
```
의존성을 바꾼 뒤 같은 빌드 오류가 반복되면 캐시를 삭제합니다.
```

→ 모호한 권고를 실행 조건으로 바꿈.

---

### 17. 책임 회피형 표현 — `검토할 수 있습니다 / 고려할 수 있습니다`

판단을 미루는 문장. 실무 문서에서는 결정 기준이 필요함.

**Before**
```
API 클라이언트 분리를 고려할 수 있습니다.
```

**After**
```
외부 API 호출이 3개 이상이면 API 클라이언트를 별도 모듈로 분리합니다.
```

→ "고려"를 임계값과 결정 규칙으로 치환.

---

### 18. 명사화 과잉 — `개선을 위한 검토를 수행`

동사를 명사로 바꾸면서 문장이 딱딱해짐.

**Before**
```
사용성 개선을 위한 UI 구조 재검토를 수행합니다.
```

**After**
```
UI 구조를 다시 보고, 클릭 수가 많은 화면부터 줄입니다.
```

→ 명사형 표현을 동사로 되돌림.

---

### 19. "제공합니다" 남발

SaaS 소개문에서 자주 보이는 표현. 제공한다는 말보다 실제 동작이 중요함.

**Before**
```
x-build는 자동 백업 기능을 제공합니다.
```

**After**
```
x-build는 매일 오전 3시에 DB 스냅샷을 저장합니다.
```

→ "기능 제공"을 구체적인 실행 방식으로 변경.

---

### 20. 결론 미루기 — `그렇다면 무엇이 적합할까요?`

글의 긴장감을 만들려다 핵심 답변이 늦어짐.

**Before**
```
그렇다면 어떤 방식이 가장 적합할까요? 이를 판단하기 위해서는 먼저 각
방식의 특징을 살펴볼 필요가 있습니다.
```

**After**
```
작은 팀은 Markdown으로 충분합니다. 검증과 조회가 필요하면 JSON Schema나
DB형 문서가 낫습니다.
```

→ 답을 먼저 말하고, 설명은 뒤에 배치.

---

### 21. 과잉 완곡어 — `보입니다 / 생각됩니다 / 판단됩니다`

확신이 필요한 문서에서 문장이 흐려짐.

**Before**
```
이 방식이 현재 구조에는 더 적합할 것으로 보입니다.
```

**After**
```
이 방식이 현재 구조에 더 적합합니다. 설정 파일 수를 줄이고 배포 절차도
단순해집니다.
```

→ 완곡어를 줄이고, 이유를 붙임.

---

### 22. 스코프 없는 최적화 — `성능을 최적화합니다`

무엇을 최적화하는지 알 수 없음.

**Before**
```
시스템 성능을 최적화해야 합니다.
```

**After**
```
이미지 압축, DB 인덱스, 캐시 TTL 세 가지를 조정합니다.
```

→ "최적화"를 조정 대상 목록으로 분해.

---

### 23. 목적어 없는 연동 — `다양한 시스템과 연동`

무엇과 연결되는지 없으면 의미가 약함.

**Before**
```
이 서비스는 다양한 외부 시스템과 연동됩니다.
```

**After**
```
이 서비스는 Slack 알림, GitHub Actions, S3 백업과 연결됩니다.
```

→ "다양한 시스템"을 실제 대상명으로 치환.

---

### 24. 인과관계 흐림 — `이를 통해 / 이로 인해`

문장 연결은 되지만, 원인과 결과가 선명하지 않음.

**Before**
```
이를 통해 운영 효율을 높일 수 있습니다.
```

**After**
```
배포 로그가 자동으로 저장되므로, 장애 발생 시 변경 지점을 바로 확인할 수
있습니다.
```

→ "이를 통해" 뒤에 숨은 실제 인과를 복원.

---

### 25. 과한 메타 설명 — `이 글에서는 / 본 문서에서는`

문서 자체를 설명하는 문장이 많아지면 본론이 늦어짐.

**Before**
```
이 글에서는 x-build의 주요 특징과 활용 방법에 대해 설명합니다.
```

**After**
```
x-build는 빌드 설정을 phase 단위로 나누고, 각 phase의 실행 순서를 고정합니다.
```

→ "무엇을 설명할지"보다 "무엇인지"를 먼저 제시.

## 공통 원칙 표 (25종 요약)

| # | 패턴 | 진단 신호 | 치환 |
|---|---|---|---|
| 1 | 가짜 균형 | `뿐만 아니라`, `~과 ~까지` | 수치·명사로 분해 |
| 2 | 결론 없는 권고 | `균형`, `상황에 따라` | 구체 분기 + fallback |
| 3 | 강조 인플레이션 | `매우/완벽/혁신/강력` | 형용사 삭제, 숫자 추가 |
| 4 | 권고형 누적 | `~해야`가 3회 이상 | 명사구 리스트 |
| 5 | 영어식 직역 | `이러한 점에서`, `~함에 있어` | 통째로 삭제 |
| 6 | 추상 효과어 | `효율성, 생산성, 사용자 경험, 품질` | 실제 변화·동작·수치 |
| 7 | 가능성 남발 | `~할 수 있습니다, ~가능합니다` | 조건 + 결과 |
| 8 | 수동태 | `~됩니다, ~제공됩니다, ~처리됩니다` | 행위자 + 동사 |
| 9 | 목차 예고문 | `먼저 살펴보고, 다음으로 알아보겠습니다` | 명사구 목차 |
| 10 | 중요합니다 엔딩 | `~하는 것이 중요합니다` | 실행 기준 |
| 11 | 포괄어 남용 | `다양한, 여러, 폭넓은, 많은` | 구체 항목 나열 |
| 12 | 근거 없는 강점 | `업계 표준, 널리 사용, 뛰어난` | 기능·근거·수치 |
| 13 | 동어 반복 | `효율적 관리, 체계적 운영` | 하나의 구조로 압축 |
| 14 | 부사어 패딩 | `기본적으로, 일반적으로, 상대적으로` | 삭제 후 근거 제시 |
| 15 | 문제 해결 클리셰 | `기존 문제를 해결합니다` | Before/After |
| 16 | 조건 없는 권고 | `필요하다면, 상황에 따라, 적절히` | 실행 조건 |
| 17 | 책임 회피형 표현 | `검토할 수 있음, 고려할 수 있음` | 임계값 + 결정 규칙 |
| 18 | 명사화 과잉 | `개선을 위한 검토 수행` | 동사형 문장 |
| 19 | 제공합니다 남발 | `기능을 제공합니다` | 실제 동작 |
| 20 | 결론 미루기 | `그렇다면 무엇이 적합할까요?` | 답 먼저, 설명 나중 |
| 21 | 과잉 완곡어 | `보입니다, 생각됩니다, 판단됩니다` | 주장 + 이유 |
| 22 | 스코프 없는 최적화 | `성능 최적화, 구조 개선` | 조정 대상 명시 |
| 23 | 목적어 없는 연동 | `다양한 시스템과 연동` | 연동 대상명 |
| 24 | 인과관계 흐림 | `이를 통해, 이로 인해` | 원인 + 결과 재작성 |
| 25 | 메타 설명 과잉 | `이 글에서는, 본 문서에서는` | 본론 문장으로 시작 |

## 적용 우선순위

편집할 때는 아래 순서로 본다.

1. **삭제** — 이러한 점에서, 기본적으로, 매우, 중요합니다
2. **구체화** — 다양한 → 실제 항목
3. **행위자 복원** — 처리됩니다 → 서버가 처리합니다
4. **조건 추가** — 필요하다면 → 어떤 경우에
5. **수치화** — 빠르게 → 3초 안에, 5단계에서 2단계로
6. **구조화** — 긴 권고문 → 명사구 리스트
7. **결론 선배치** — 설명보다 판단을 먼저 제시
